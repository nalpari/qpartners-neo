🔴 HIGH 심각도즉시 조치 권장

독립 실행된 2개 감사 에이전트가 동일하게 지목했고, 미들웨어·인증 헬퍼 코드를 직접 정독하여 확정한 발견입니다.

VULN-01 미들웨어가 public GET 경로에서 인바운드 X-User-* 헤더를 제거하지 않아 익명 사용자가 관리자 권한 위조 가능
High broken_access_control 신뢰도 8/10
위치
src/middleware.ts:115  src/lib/auth.ts:67-86
설명
전체 인증 모델은 "X-User-Type/Id/Role 헤더는 오직 미들웨어가 JWT 검증 후 주입한다(신뢰값)"는 전제 위에 서 있습니다(auth.ts:1-7 주석이 "Gateway 없이 직접 노출할 경우 반드시 JWT 검증 로직을 추가해야 한다"고 경고). 그러나 미들웨어는 클라이언트가 보낸 인바운드 X-User-* 헤더를 명시적으로 제거하지 않습니다. PUBLIC_GET_PATTERNS에 해당하는 GET 요청에서 쿠키 토큰이 없거나/만료/2FA 미완료인 경우, 코드는 return NextResponse.next()(115행)로 빠지며 원본 요청 헤더를 그대로 핸들러에 전달합니다. getUserFromHeaders는 이 헤더를 HMAC/서명 검증 없이 형식만 확인하므로, 핸들러는 위조된 권한을 신뢰합니다.
코드 근거
// middleware.ts — public GET 비회원/무효토큰 시 원본 헤더 그대로 통과
if (isPublicGet) {
  const publicToken = request.cookies.get(COOKIE_NAME)?.value;
  if (publicToken) { /* ...유효 시에만 X-User-* set... */ }
}
return NextResponse.next();   // 115행: 클라이언트 X-User-* 미제거

// auth.ts:72 — 헤더 무검증 신뢰
if (!userType || !userId || !role) return null;   // 형식만 확인
익스플로잇 시나리오
쿠키 없이 위조 헤더를 붙인 익명 요청:
GET /api/contents/5/files/10/download
X-User-Type: ADMIN / X-User-Role: SUPER_ADMIN / X-User-Id: x

미들웨어는 isPublicGet=true + 무쿠키 → 115행 통과(헤더 미제거). 핸들러의 isInternalUser("SUPER_ADMIN")===true(contents/route.ts:60)가 게시대상(roleCode)·게시기간 필터를 무시 → 미게시(draft)·소프트삭제(deleted)·특정 파트너 전용 콘텐츠와 첨부파일·인라인 이미지를 비인증 열람. 동일 위장으로 GET /api/contents(목록), /api/contents/[id], download-all(ZIP), /api/inline-images/[id], /api/home-notices/active까지 확장됩니다.
배포 구조와 위험도
운영은 network_mode: host로 Next.js standalone이 포트 5000/5001에 직접 노출됩니다(docker-compose-production.yml). 리포지토리에 인바운드 헤더를 거르는 nginx/게이트웨이 설정 파일이 존재하지 않으며, next.config.ts에도 헤더 스트리핑이 없습니다. 따라서 미들웨어가 사실상 유일한 보안 경계입니다. 만약 앞단 프록시가 X-User-*를 제거하도록 설정되어 있다면 실제 악용은 차단되지만, 이는 코드가 secure-by-default가 아니며 인프라 설정 1건에 전적으로 의존함을 의미합니다 — 코드 레벨 결함은 명백합니다.
권장 조치
미들웨어 최상단에서 모든 분기 이전에 인바운드 X-User-Type/Id/Role/Name/Department를 무조건 삭제한 새 Headers를 만든 뒤, 인증된 경우에만 서버 값으로 .set()하세요. public GET 비인증 fall-through(115행)도 sanitized 헤더로 NextResponse.next({ request: { headers } })를 반환해야 합니다. 추가 방어로 주입 헤더에 요청별 공유 시크릿/서명을 부여하면 프록시 미스컨피그에도 견고해집니다.

VULN-03 자동로그인 cipher 24시간 재생(replay) + ADMIN/SUPER_ADMIN 2FA 면제
Medium crypto / replay 신뢰도 7/10
위치
src/lib/auto-login-crypto.ts:28-37  src/app/api/auth/auto-login/inbound/route.ts
설명
Inbound 자동로그인은 AES-128-CBC(비인증 암호화, 인증태그 없음) + 결정적 IV(YYYYMMDD(KST)+_autoL!!)를 사용해, 같은 사용자·같은 날 cipher가 항상 동일합니다. 받는 측 일회용 소진 차단이 의도적으로 제거되어 cipher 1건이 자정까지 무제한 재사용 가능합니다. 더불어 cipher 검증만으로 세션을 발급하며 twoFactorVerified: detail.pwdInitYn !== "N"로 설정 — 즉 ADMIN/SUPER_ADMIN 포함 전원이 2FA를 면제받습니다. cipher는 GET 쿼리스트링으로 전달되어 브라우저 히스토리·Referer·프록시 액세스 로그에 평문 잔존하므로 탈취 표면이 넓습니다.
코드 근거 (코드 주석이 위험을 명시)
// auto-login-crypto.ts:30-33
// IV 가 (KST 일자 + 고정 상수) 로 결정적 → 같은 userId 가 같은 날 동일 cipher
// 받는 측 1회용 차단을 두지 않음 ...
// 받아들인 위험: cipher 탈취 시 24h 내 재사용 가능
// inbound/route.ts
twoFactorVerified: detail.pwdInitYn !== "N",   // ADMIN 포함 2FA 면제
익스플로잇
공격자가 공유 PC 브라우저 히스토리/프록시 로그/Referer에서 ADMIN의 autoLoginParam1 cipher를 1회 획득 → 같은 날 자정까지 /api/auth/auto-login/inbound?autoLoginParam1=<cipher>로 재전송 → 비밀번호·2FA 없이 ADMIN 세션(8h JWT) 획득.
맥락 — 의도된 설계 트레이드오프
코드 주석에 따르면 이는 외부 3사(QSP/Q.Order/Q.Musubi) 호환을 위한 의도적 결정이며 위험을 인지·수용한 상태입니다. 따라서 "결함"이 아닌 "문서화된 잔존 위험"으로 분류하되, 권한 상승 경로(ADMIN 무비밀번호 진입)라는 점에서 추가 가드를 강력 권고합니다.
권장 조치
최소한 ADMIN/SUPER_ADMIN 응답 시에는 자동로그인을 거부하거나 2FA를 강제하세요. 가능하면 평문에 nonce/타임스탬프를 포함해 inbound 측 일회용 소진(used 마킹)을 도입하거나, GCM 등 인증 암호화 + 짧은 만료로 전환합니다(주석에 "사양 확장으로 강화 가능"으로 명시됨).
VULN-04 비밀번호 재설정 요청에서 사용자 열거(user enumeration) — 200 vs 404 구분
Medium user_enumeration 신뢰도 8/10
위치
src/app/api/auth/password-reset/request/route.ts:327-338
설명
회원이 존재하면 HTTP 200 { data: { message } }, 미존재/매칭실패면 HTTP 404 { error: "一致する会員情報がありません" }를 반환합니다. 상태 코드와 본문 메시지가 모두 달라 공격자가 임의 이메일/loginId의 가입 여부를 정확히 판별할 수 있습니다. 이는 프로젝트 자체 규칙(.claude/rules/api.md: "회원 존재 여부를 응답으로 구분 불가하게 설계 — 동일 200 반환")과 정면 충돌합니다. (코드는 not-found/mismatch/ambiguous "끼리"는 동일 메시지로 통일했으나, 존재 케이스와는 여전히 구분됩니다.)
코드 근거
// 미존재 경로 (404)
return NextResponse.json({ error: "一致する会員情報がありません..." }, { status: 404 });
// 존재+발송 경로 (200)
return NextResponse.json({ data: { message: "...送信しました。" } });
익스플로잇
공격자가 표적 이메일 목록을 전송 → 404면 미가입, 200이면 가입 회원으로 분류. 수집된 유효 계정 목록은 credential stuffing·피싱 표적이 됩니다.
권장 조치
존재/미존재/형식오류를 모두 동일하게 200 + 동일 메시지("해당 정보로 가입된 계정이 있으면 메일을 보냈습니다")로 응답하고, 실제 발송 여부는 서버 내부에서만 분기하세요.
VULN-05 로그인 라우트에 브루트포스/credential-stuffing 방어 부재
Medium brute_force 신뢰도 7/10
위치
src/app/api/auth/login/route.ts (rate limit 호출 전무)
설명
password-reset·email/check·password-init·auto-login/inbound 등 거의 모든 인증 라우트에 checkRateLimit이 적용돼 있으나, 정작 핵심인 /api/auth/login에는 IP·계정 기반 rate limit이 전혀 없습니다. loginId+pwd를 무제한으로 QSP에 프록시하며, QSP 응답의 loginFailCnt 필드도 읽거나 강제하지 않습니다. 따라서 우리 계층의 시도 제한은 0이며 외부 QSP 잠금 정책에만 의존합니다. signup도 동일하게 rate limit이 없어 메일 발송 남용 표면이 있습니다.
익스플로잇
단일 IP에서 한 계정에 비밀번호 사전 공격을 무제한 시도(QSP 잠금 임계가 높으면 성공 가능), 또는 다수 계정에 흔한 비밀번호로 credential stuffing — 우리 서버가 QSP에 대한 증폭 프록시 역할을 합니다.
권장 조치
다른 라우트와 동일하게 IP 기반 + account:loginId fallback rate limit을 로그인 진입 직후 적용하세요. QSP loginFailCnt를 응답에 반영해 잠금 상태를 우리 계층에서도 인지합니다. (단, 아래 LOW-09의 다중 인스턴스·XFF 한계와 함께 인프라 레벨 WAF 병행을 권장.)

VULN-07 보안 응답 헤더 전역 부재 — CSP · X-Frame-Options · HSTS · Referrer-Policy
Medium missing_headers 신뢰도 8/10
위치
next.config.ts (headers() 미정의)  src/middleware.ts:197 (matcher /api/* 한정)
설명
next.config.ts에 headers() 설정이 전혀 없고, 미들웨어 matcher는 /api/:path* 한정이라 보안 헤더를 추가하지 않습니다. 따라서 모든 HTML 페이지가 CSP / X-Frame-Options / HSTS / 전역 X-Content-Type-Options / Referrer-Policy 없이 서빙됩니다. 이 앱은 BlockNote/Tiptap 리치 에디터로 작성된 사용자 HTML을 저장·렌더하므로(DOMPurify 1차 방어는 존재), sanitizer 우회가 발견되면 CSP 부재 시 즉시 임의 스크립트 실행으로 직결됩니다. 비인증 페이지(로그인·회원가입·문의)는 X-Frame-Options 부재로 클릭재킹/UI redress 표면도 노출됩니다.
코드 근거
// next.config.ts — headers() 정의 없음
const nextConfig: NextConfig = {
  reactCompiler: true,
  output: "standalone",
  // async headers() { ... } 부재 → CSP/XFO/HSTS 전무
