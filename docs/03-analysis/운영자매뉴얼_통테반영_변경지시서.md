**(2) 소스코드 수정 — 완료 (2026-05-27)**

원래 매뉴얼은 양방향으로 구현과 어긋나 있었음:
- 종료일: 매뉴얼 "미입력 가능" ↔ 구현은 **필수** (`home-notice.ts:26-27` create 스키마, DB `endAt DateTime` non-null, 팝업 `validate()` line 134). → 구현이 스펙과 일치, **변경 불요**.
- 시작일: 매뉴얼 "Today 기본값" ↔ 구현은 **기본값 없음(null)** (`notice-form-popup.tsx` 시작일 `useState` 가 `parseDate("")`=null 로 초기화). → **불일치, 코드 수정함.**

수정 내용: `src/components/popup/notice-form-popup.tsx` — 신규 등록(create) 모드에서 시작일 기본값을 **Today** 로 초기화 (lazy initializer, edit 모드는 기존값 유지).
```tsx
const [startDate, setStartDate] = useState<Date | null>(() => {
  const parsed = parseDate(initialData?.startDate ?? "");
  // 신규 등록은 emptyForm(빈 startDate "")을 notice 로 넘기므로 initialData 존재 여부가 아닌 mode 로 분기.
  return parsed ?? (mode === "create" ? new Date() : null);
});
```
> 주의: 등록 핸들러(`notices-table.tsx:343` handleRegister)가 등록 모드에서도 `startDate:""` 인 `emptyForm` 을 `notice` 로 전달함. 따라서 `initialData` 존재 여부가 아니라 **`mode`** 로 분기해야 Today 기본값이 적용됨.

> 비고: #2258(종료일 NULL)은 **[콘텐츠]** 전용이며 홈공지에는 적용 대상 아님. 홈공지는 시작·종료 모두 필수가 확정 스펙.

## 3. Part C — 콘텐츠 게시 종료일 NULL 처리: 기재할 description (#2258)

콘텐츠(`ContentTarget`)는 **시작·종료 모두 선택 입력**이며, 종료일 미입력 = 종료 제한 없는 상시 게시다.
- 근거: `src/lib/schemas/content.ts:28-29` (`startAt`/`endAt` `.optional()`), `prisma` `ContentTarget.startAt/endAt DateTime?`, `computeStatus` 로 상시/예정/종료 판정.

### 3-1. 테이블정의서 — 기재 완료 (추가 불요)
`테이블정의서_QP_TO-BE_v1.0.xlsx` `qp_content_targets` 시트 `end_at` 설명에 이미 기재됨:
> `게시 종료일. NULL=종료일 제한 없음 (p.25 #8)`

### 3-2. 운영자매뉴얼(콘텐츠 신규섹션) / 화면설계서 — 아래 문구 사용
콘텐츠 등록 화면이 매뉴얼에 없으므로, 신규 섹션 작성 시 게시기간 항목 설명에 추가:

- **화면 설명(※ 주석, 한국어):**
  > ※ 게시 시작일·종료일은 선택 입력입니다. 종료일을 미입력하면 종료일 제한 없이 상시 게시(상시 공개)됩니다.
- **UI 보조문구(일본어, 종료일 입력란 근처):**
  > 未入力の場合は終了日の制限なく常時掲載されます。

### 3-3. FLAG-2 (확인 필요) — #2258 "시작일 필수" 미구현 의심
#2258 항목3은 `[시작일] 필수 입력`을 요구하나, `content.ts:28` 의 `startAt` 은 `.optional()` 로 **시작일도 선택**이다.
→ 시작일 필수가 UI 단에서만 강제되는지, 아직 미반영인지 **개발/QA 확인 필요**. (미반영이면 별건 코드 수정)
